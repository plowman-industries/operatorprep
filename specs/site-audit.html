P0 · Blocking Issues

Fix these first. They break the experience, the brand, or the site.

A-01

Footer renders twice — Astra default stacked on custom footer

P0

Observed

The live site renders two footers stacked on top of each other. First the default Astra theme footer ("Copyright © 2026 Operator Prep | Powered by Astra WordPress Theme"), then the custom OperatorPrep footer with the four-column layout.

Why it matters

It makes the site look unfinished. It damages brand credibility. And the "Powered by Astra" line is literally free advertising for a theme vendor on your page.

Fix

  1. Log into WordPress admin.
  2. Go to Appearance → Customize → Footer Builder.
  3. In the Footer Bottom row, click the "Copyright" module and remove it.
  4. If using Astra Pro: Appearance → Customize → Footer → Footer Layout → set to "Disabled".
  5. Fallback (if the above doesn't work): add .ast-small-footer { display: none !important; } under Appearance → Customize → Additional CSS.
  6. Publish and verify only one footer renders on the home page and an interior page.
A-02

Homepage still uses emojis instead of custom SVG icons

P0

Observed

The live home page at operatorprep.com still renders emojis: 🎖️ 💧 ⚡ 📍 📝 🃏 🔢 📖 🔧 ⚙️. The design system has required custom SVG icons since v1.1 (February).

Why it matters

Emojis render differently across OS/browser — Apple's 💧 is a 3D blue drop, Google's is a flat teal drop, Windows' is something else entirely. The brand loses control of its visual language every time someone visits from a different platform. Also: the brand guidelines explicitly say "no emojis" and show a DON'T example for exactly this drop emoji.

Fix

  1. Identify all homepage blocks using emojis. There are two sections: Credentials (🎖️💧⚡📍), What's Included (📝🃏🔢📖), and Certifications Covered (💧🔧⚙️).
  2. Replace each emoji with the corresponding SVG from the design-system sprite. Mapping: 🎖️→i-certified, 💧→i-droplet, ⚡→i-power, 📍→i-location, 📝→i-test, 🃏→i-flashcard, 🔢→i-math, 📖→i-guide, 🔧→i-pipe, ⚙️→i-basin.
  3. Paste the SVG sprite from the design system into the WP theme's header.php or a global HTML block so <use href="#i-xxx"> works sitewide.
  4. Update the homepage Custom HTML blocks to use <svg><use href="#i-droplet"/></svg> patterns instead of emoji characters.
  5. Audit every /t*/, /d*/, /ww*/ grade landing page for the same issue.
A-03

Footer links to /certification/t1 — those URLs don't exist

P0

Observed

The footer's "Certifications" column links use the URL pattern /certification/t1, /certification/d1, /certification/ww1. The live site uses /t1/, /d1/, /ww1/. The /certification/* URLs 404.

Why it matters

Every visitor who clicks a footer link lands on a 404. This is the primary navigation to the revenue-generating pages, and it's broken.

Fix

  1. Edit the footer HTML block in WordPress (likely in Appearance → Widgets or a footer HTML module).
  2. Change href="https://operatorprep.com/certification/t1" → href="https://operatorprep.com/t1/".
  3. Apply the same change for D1 and WW1. Or change these to link to the track overview pages if those exist.
  4. Run a sitewide link checker (e.g. Broken Link Checker plugin) to find any other instances of the wrong pattern.

P1 · Major Issues

Fix these in the same sprint. They violate the design system in ways users see.

A-04

Header uses old PNG logo banner instead of clean SVG brand mark

P1

Observed

Header renders operatorprep-logo-banner.png from /wp-content/uploads/2026/03/. This is a raster PNG with a baked-in dark background. Scales poorly on retina, can't be recolored.

Why it matters

PNG logos are blurry on high-DPI screens. They can't adapt to dark/light contexts. Logo refresh work has already produced clean SVG versions that scale infinitely and use currentColor.

Fix

  1. Upload the SVG logo assets to /wp-content/uploads/.
  2. Replace the header logo in Appearance → Customize → Site Identity with the SVG.
  3. If Astra doesn't allow SVG upload by default, add the Safe SVG plugin first.
  4. Test across light pages, dark pages, and retina/mobile displays.
A-05

"Certifications" and "Pricing" nav links both go to /pricing/

P1

Observed

The primary nav has six items: Home, Certifications, Pricing, About, FAQ, My Account. "Certifications" and "Pricing" both link to operatorprep.com/pricing/. That's two nav slots doing the same job.

Why it matters

Wastes prime real estate in the nav. Confuses users ("what's the difference?"). Likely signals to search engines that the site structure isn't well thought out.

Fix — choose one:

  1. Option A: Build a dedicated /certifications/ overview page listing all 15 certs grouped by track. Have "Certifications" point there; "Pricing" continues to point to /pricing/.
  2. Option B: Drop "Certifications" from the nav entirely — use the freed slot for "My Dashboard".
  3. Recommended: Option A. Certifications page becomes the natural entry point for organic search traffic, pricing page is the conversion step.
A-06

Duplicate T4 Flashcards page: /t4-flashcards/ and /t4-flashcards-2/

P1

Observed

Sitemap contains both /t4-flashcards/ and /t4-flashcards-2/. WordPress auto-appends -2 when a page with the same slug already exists — usually means a duplicate was created by accident and never deleted.

Why it matters

SEO duplicate content. Split traffic/analytics. Risk that the subscription gating is configured on one but not the other.

Fix

  1. In WP admin: Pages → All Pages, filter by "t4 flashcards".
  2. Determine which page has current content / is linked from other pages / has the right subscription gating. Keep that one.
  3. Delete the other. Or set up a 301 redirect from the deleted slug to the kept slug using the Redirection plugin.
  4. Audit the other T-grades, D-grades, WW-grades for similar duplicates.
A-07

Practice-test pages use dark body — hierarchy inverted

P1

Observed

On the T2 Practice Tests page, the body uses a dark navy background. The primary CTA (Practice Test card) fades into the dark background while secondary topic cards dominate visually. The mastery legend is nearly invisible.

Why it matters

Inverted visual hierarchy. Users looking for the "start a quiz" primary action see the topic cards first. This is the exact issue that prompted design system v1.3.

Fix

  1. Rebuild the practice-test template HTML using the v1.3 component classes.
  2. Set the page body background to --color-mist (light), NOT dark ink.
  3. The Practice Test hero card handles the dark ink treatment — not the whole page.
  4. Apply the same template to all 15 practice-test pages, all 15 flashcard pages, all 15 math-drill pages.
  5. See design system v1.3, Section 04 Page Templates, for the locked anatomy.

P2 · Minor Issues

A-08

Sitemap exposes all 90+ URLs as a flat list in the footer source

P2

Observed

Crawling the home page reveals an auto-generated list of every single URL on the site in the rendered HTML, not behind a toggle.

Why it matters

HTML clutter that doesn't help users and may hurt SEO by flattening link equity across too many pages.

Fix

  1. Identify which WP widget or plugin is generating this list. Likely "All Pages" widget or a sitemap widget in a footer area.
  2. Remove the widget. Rely on the proper footer columns for human navigation.
  3. Keep XML sitemap at /sitemap.xml for search engines.
A-09

Inconsistent page coverage across grades

P2

Observed

The 4-study-tool grid isn't evenly present across all 15 grades:

  • T5: has Flashcards, Practice Tests, Quick Reference, Study Guide — but NO Math Drills page.
  • WW5: has Flashcards, Practice Tests, Quick Reference, Study Guide — but NO Math Drills page.
  • D5: has all four.
  • T3 and T4: have Shift Scenario Simulator; others don't.

Why it matters

Users on T5 who click "Math Drills" will hit a 404 or an empty state.

Fix

  1. Build a coverage matrix: 15 grades × 4 tools = 60 pages. Mark which exist and which don't.
  2. For gaps, either: (a) build the missing pages, or (b) hide the tool link on the dashboard for grades where it doesn't exist.
  3. Document the supported tool matrix in the design system so new grades always get full coverage.
A-10

"Powered by Astra WordPress Theme" in footer copyright line

P2

Observed

The duplicate Astra footer includes a "Powered by Astra WordPress Theme" hyperlink. Free ad space given to the theme vendor.

Why it matters

Fix

Resolved as part of A-01. Once the duplicate Astra footer is disabled, this disappears with it.

P3 · Polish

A-11

Empty <h1> tag at top of homepage before real H1

P3

Observed

The homepage HTML contains an empty H1 (rendered as blank) before the intended H1 "Pass Your Certification. Know Your Plant." This is probably Astra's default page-title block not being hidden.

Why it matters

Semantic HTML violation. Screen readers announce an empty heading. SEO bots may interpret the real H1 as H2 or miss it.

Fix

  1. In WP Customizer: Appearance → Customize → Layout → Containers → Page Title. Disable the default page title for the home page.
  2. Alternative CSS fix: body.home .entry-title { display: none; } in Additional CSS.
A-12

"Shopping Cart" text appears in page body randomly

P3

Observed

Crawled home page has the literal text "Shopping Cart" rendered in the page body, floating above the footer. Likely a WooCommerce widget leaking into a sidebar area.

Why it matters

Looks like a rendering error.

Fix

  1. Appearance → Widgets. Find the "Shopping Cart" widget in any active widget area.
  2. Remove it. The cart is accessible via /cart/ or a nav cart icon if you add one.
